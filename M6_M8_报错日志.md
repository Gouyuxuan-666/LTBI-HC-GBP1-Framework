## M7: 分子动力学模拟 (GROMACS 2020.6)

### Issue #1: Python timeout 杀死了 50ns Production MD (CRITICAL)

**时间**: 2026-07-05 22:10 ~ 2026-07-07 06:28

**错误信息**:
```
subprocess.TimeoutExpired: Command 'cmd.exe /c "gmx mdrun -deffnm prod -v -ntmpi 1 -gpu_id 0"' timed out after 3600 seconds
```

**原因**: `M7_md.py` 中 `run_gmx()` 函数默认 `timeout=3600`（1小时），但50ns MD在GTX 1080上以41.4 ns/day速率需要约29小时。Python的`subprocess.run`在超时后杀死了cmd.exe进程。

**关键发现**: 实际上 mdrun 进程并未被真正杀死。Windows上杀死cmd.exe不会连带杀死GROMACS子进程，导致我们误以为需要完全重跑。mdrun实际上在后台默默跑完了全部50ns（实际耗时32小时，37.3 ns/day），完成于2026-07-07 06:28。

**修复**:
1. `run_production()` 中生产跑的 `run_gmx(..., timeout=None)` — 取消超时
2. 添加 `-cpi prod.cpt` checkpoint 续跑逻辑
3. 添加 `--continue` 命令行参数：跳过 EM/NVT/NPT 直接续跑
4. grompp 跳过逻辑：若 `prod.cpt` 已存在则跳过 grompp（TPR必须匹配checkpoint）

### Issue #2: 分析步骤只有 xvg 数据，没有可视化PDF (MAJOR)

**原因**: 原始 `analyze()` 函数只调用 `gmx rms/rmsf/gyrate/sasa` 输出 `.xvg` 文件，没有画图。

**修复**:
1. 在 `M7_md.py` 的 `analyze()` 中添加 `parse_xvg()` + `plot_figure()` 使用 matplotlib 生成 PDF
2. 额外创建独立脚本 `M7_plot.py`：纯分析+画图，不依赖完整MD管线
3. 添加 `--analyze` 命令行参数
4. 输出6个PDF：rmsd.pdf, rmsf.pdf, rg.pdf, sasa.pdf, hbonds.pdf, M7_QC_panel.pdf

### Issue #3: M7_plot.py 在 RMSF 图处 PermissionError (MINOR)

**错误信息**:
```
PermissionError: [Errno 13] Permission denied: 'D:\\s11\\M7_output\\rmsf.pdf'
```

**原因**: Windows 文件锁竞争。前一个 PDF (rmsd.pdf) 刚写入时可能被杀毒软件/索引服务短暂锁定，导致下一个PDF写入失败。

**修复**: `plot_figure()` 改为先保存到 `.tmp` 文件，再 `unlink()` + `rename()` 覆盖目标。

### Issue #4: 氢键分析 group selection 失败 (MINOR)

**原因**: GROMACS `hbond` 需要选择两组（供体和受体）。`run_gmx()` 的 `stdin_input` 用 `echo` 管道传参，对多行输入（如 "1\n14\n"）支持不好——cmd.exe的echo无法正确处理换行。

**修复**: 重写 `run_gmx()`：多行stdin改为写入临时文件，用 `cmd.exe /c "type tmpfile | gmx ..."` 管道输入。避免了echo的换行问题。

### Issue #5: H键分析结果为0（非报错，但需注意）

**结果**: mean = 0.00 ± 0.02

**解释**: 不是bug。白藜芦醇的二苯乙烯骨架靠π-π堆积+疏水作用嵌入GBP1口袋，3个酚羟基朝向溶剂而非蛋白内部。与对接预测（vdW主导，无极性接触）和MMPBSA结果（ΔE_elec ≈ -8 kJ/mol，约为vdW的5%）一致。

## M8: MMPBSA 结合自由能计算

### Issue #6: gmx_MMPBSA 安装失败链 (CRITICAL)

**涉及包**: gmx_mmpbsa 1.6.5

**依赖冲突**:
| 包 | 要求版本 | 已安装版本 | 解决方式 |
|------|---------|-----------|---------|
| numpy | ==1.26.4 | 2.5.0 | 不兼容，需预编译wheel |
| matplotlib | ==3.7.3 | 3.11.0 | --no-deps跳过 |
| pandas | ==1.5.3 | 3.0.3 | 安装3.0.3后忽略 |
| scipy | ==1.14.1 | 1.18.0 | --no-deps跳过 |
| seaborn | ==0.11.2 | 0.13.2 | 安装0.13.2后忽略 |
| mpi4py | ==4.0.1 | 4.1.2 | 需MS-MPI库 |
| parmed | >=4.2.2 | 未安装 | 需C++编译器编译 |

**安装过程**:
1. `pip install gmx_mmpbsa` → numpy==1.26.4 从源码编译失败（无C编译器）
2. `pip install numpy`（预编译版2.5.0 OK）→ 但gmx_mmpbsa仍要numpy==1.26.4
3. `pip install gmx_mmpbsa --no-deps` → 成功安装但缺少所有依赖
4. `pip install parmed` → 失败：`Microsoft Visual C++ 14.0 or greater is required`
5. 安装 VS Build Tools 2022 (~3GB) → 编译器就绪
6. `pip install parmed seaborn mpi4py` → 成功安装
7. `gmx_MMPBSA --version` → 失败：`cannot load MPI library / msmpi.dll`
8. 从Microsoft CDN下载 MS-MPI v10.1 安装包 → 安装成功
9. `gmx_MMPBSA --version` → **终于成功!** `gmx_MMPBSA.exe 1.6.5`

### Issue #7: gmx_MMPBSA 输入变量名错误 (MAJOR)

**错误1**: `InputError: Unknown variable forcefield in &general`
- 修复：删除 `forcefield="ff99SB-ildn"` 参数（v1.6.5不支持）

**错误2**: `InputError: Unknown variable strip_mask in &general`
- 修复：删除 `strip_mask`, `receptor_mask`, `ligand_mask` 参数（v1.6.5不支持）

**最终方法**: 用 `-ci complex.ndx -cg group1 group2` 命令行传参代替namelist变量

### Issue #8: gmx_MMPBSA 缺少 cpptraj (CRITICAL — 最终放弃gmx_MMPBSA)

**错误信息**:
```
MMPBSA_Error: Could not find necessary program [cpptraj]
```

**原因**: gmx_MMPBSA 底层依赖 AmberTools 的 `cpptraj` 进行轨迹处理。服务器未安装AmberTools。

**决定**: 放弃 gmx_MMPBSA，改用纯 GROMACS + Python 手动方案。无需安装任何额外包。

### Issue #9: 手动 MMPBSA 脚本开发

**方案**: `M8_mmpbsa.py` — 纯 GROMACS + Python (numpy + matplotlib only)

**流程**:
1. `gmx trjconv` — 从70GB prod.xtc中提取100帧 (protein+UNL only, 去水)
2. `gmx grompp` — 用含 `energygrps = Protein UNL` 的MDP生成rerun TPR
3. `gmx mdrun -rerun` — 对100帧重算能量（单点能，极快）
4. `gmx energy` — 从rerun.edr提取 LJ-SR:Protein-UNL 和 Coul-SR:Protein-UNL
5. Python 计算 ΔE_vdw + ΔE_elec + 估算的 ΔG_GB + ΔG_np

**结果**:
| 能量项 | 值 (kJ/mol) |
|--------|------------|
| ΔE_vdw | -168.09 ± 14.21 |
| ΔE_elec | -7.52 ± 7.86 |
| ΔG_bind (估算) | ≈ -174 kJ/mol (-41.6 kcal/mol) |

### Issue #10: MMPBSA ΔG_bind 绝对值被审稿质疑

**问题**: -41.6 kcal/mol 对蛋白-配体结合来说数值偏大（典型为-5到-12 kcal/mol）

**原因**: GB极性溶剂化项使用了过于粗略的估算（-0.5 × ΔE_elec = +3.76 kJ/mol）。完整的GB-OBC求解器会给出更大的去溶剂化惩罚（+50到+100 kJ/mol），从而产生更合理的ΔG。

**修复（论文v2）**: 改为只报告可靠的相对分量（ΔE_vdw, ΔE_elec），去掉近似估算的绝对值。添加免责声明：
> "Absolute ΔG_bind values derived from the approximate GB solvation term should be interpreted with caution; more rigorous free energy calculations are warranted."

## 总结

| 模块 | 关键报错数 | 最终解决方式 | 耗时 |
|------|-----------|-------------|------|
| M6 | 0 | — | 分钟级 |
| M7 | 5 (1 CRITICAL, 1 MAJOR, 3 MINOR) | 修复timeout + 写独立出图脚本 | MD 32h + 分析1-2h |
| M8 | 5 (3 CRITICAL, 1 MAJOR, 1 MINOR) | 放弃gmx_MMPBSA → 手动方案 | 开发2h + 计算30min |

**最大教训**: 
1. Windows + Python 3.13 环境下不要试图从源码编译科学计算包——优先用 conda 预编译二进制
2. GROMACS的`subprocess.run`超时不能可靠杀死GPU子进程——生产跑用`timeout=None`
3. 70GB MD轨迹的I/O是瓶颈——分析时优先用`-dt`抽帧模式
4. MMPBSA绝对值的可信度取决于溶剂化模型精度——简化的GB估算只适合报告相对能量贡献
